## Goal
Reduce per-command overhead in the JavaScript `redis.call`/`redis.pcall` bridge so loop-heavy scripting workloads are significantly faster while preserving Redis-compatible scripting semantics.

## Acceptance Criteria
- [ ] Eliminate avoidable per-argument UTF-8/String allocations in the bridge path (`redis_call`, `js_value_to_string`, and command argument preparation) where byte-preserving conversion is sufficient.
- [ ] Reduce temporary allocations in response conversion (`resp_to_js`) for arrays/strings without changing visible return shapes for `call` and `pcall`.
- [ ] Keep behavior compatible for error handling (`pcall` returns `{ err = ... }`, `call` throws) and type conversion expected by existing scripting tests.
- [ ] Improve throughput for scripting cases dominated by repeated `redis.call` dispatch (`hash_sum`, `set_members`, `bulk_incr`) versus current baseline logs in `tmp/lua_js_comparison_20260220_215755.txt`.

## Notes
- Hotspot evidence points to conversion overhead in `src/mini_redis/server.rs` (`redis_call`, `resp_to_js`, `js_to_resp`, `js_value_to_string`).
- Test plan:
	- Run `python3 scripts/bench_scripting.py --engine mini-redis --script-dir tests/scripting_js_faithful --iterations 1000 --warmup 200` before/after.
	- Run `cargo test --lib` and `make test-all` to guard semantic regressions.
	- Compare updated JS results against Redis Lua using `python3 scripts/bench_scripting.py --engine redis-lua --script-dir tests/scripting` and confirm targeted ratio improvements.
