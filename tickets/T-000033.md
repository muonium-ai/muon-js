## Goal
Improve throughput of the `bulk_incr` scripting benchmark (currently 0.19x vs Redis Lua) by reducing per-invocation overhead in the `redis_call()` C callback path.

## Context
The `bulk_incr` case (`tests/scripting_js_faithful/11_bulk_incr.js`) runs 102 `redis.call()` invocations per script execution (1× SET + 100× INCRBY + 1× GET). The loop body is trivial (INCRBY 1), so the overhead is entirely in the `redis_call()` callback: FFI crossing, string marshalling, per-call `block_on(mutex.lock())`, command parsing, full dispatch, and response conversion.

## Acceptance Criteria
- [ ] Reduce `redis_call()` overhead so `bulk_incr` reaches ≥0.35x vs Redis Lua (≈2× current).
- [x] No regression on other scripting cases.
- [x] All existing tests pass.

## Approach Ideas
- Hold DB lock for entire script execution (see T-000031) — eliminates 100 lock/unlock cycles.
- Fast-path SET/GET/INCRBY dispatch in `redis_call` to skip full match arm traversal.
- Avoid re-parsing command string when it's already a known uppercase command.
- Reduce `js_value_to_arc_bytes` allocation: use stack buffer for small values.

## Notes
- T-000031 implemented script-level DB lock, skip_aof, and fast-path dispatch for SET/GET/INCRBY. Result: bulk_incr improved to 0.20x (from 0.19x). Bottleneck is QuickJS interpreter overhead (~17× per-call gap vs LuaJIT). Reaching ≥0.35x requires JS engine improvements.

## Test Plan
- Run 3-round Lua-vs-JS gate before and after.
- Compare `bulk_incr` RPS improvement.
- Verify no regression on other cases.
