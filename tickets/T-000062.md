## Goal
Reduce JS-to-Redis command bridge cost so redis.call-heavy scripts are faster.

## Results (post-optimization)

Note: ratio = `js_rps / lua_rps` — values > 1.0 mean JS is faster than Lua.

| Case | Baseline | Current | Improvement |
|------|----------|---------|-------------|
| redis_call | 1.88x | 3.11x | +65% |
| incrby | 1.87x | 3.19x | +71% |
| hello | 2.07x | 3.56x | +72% |

## Changes made
- **Zero-copy byte extraction**: Added `js_value_as_bytes()` / `js_value_raw_bytes()` helpers to extract
  bytes from JS string values without mutable borrow or heap allocation
- **Integer fast path**: Added `i32_to_stack_bytes()` for direct int→bytes on stack, avoiding JS string
  creation for numeric arguments (e.g. `redis.call('SET', key, 10)`)
- **SET optimization**: Uses `set_string_from_slices()` to pass borrowed key/value to store, eliminating
  intermediate Vec allocations
- **INCRBY/INCR/GET/DEL/HGET**: Keys borrowed directly as `&[u8]` — no heap allocation
- **DECRBY fast path**: Added new fast-path dispatch for DECRBY command
- **Error helpers**: Factored out `wrongtype_error()` / `int_range_error()` to reduce code duplication
- **Statement dispatch**: Rewrote `eval_program_stmt()` with first-byte dispatch to skip keyword
  prefix chain for expression statements (e.g. `redis.call(...)` skips straight to eval_expr)
- **Command name extraction**: Uses raw byte pointer from JS heap instead of JS_ToCString
