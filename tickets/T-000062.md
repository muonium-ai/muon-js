## Goal
Reduce JS-to-Redis command bridge cost so redis.call-heavy scripts approach Lua parity

## Current state

Benchmarks that use `redis.call()` are significantly slower than Lua equivalents:
- `redis_call` (SET+GET): 2.39x slower — `redis.call('SET',k,v); return redis.call('GET',k);`
- `incrby` (SET+INCRBY): 2.31x slower — `redis.call('SET',k,10); return redis.call('INCRBY',k,v);`

The JS `redis.call()` bridge converts JS values → Rust strings → command dispatch → Rust
values → JS values. Each call likely involves multiple string allocations and Value conversions.

## Acceptance Criteria
- [ ] `redis_call` benchmark ratio drops below 1.8x
- [ ] `incrby` benchmark ratio drops below 1.8x
- [ ] `make lua-js-perf-check` passes gate
- [ ] All cargo tests pass, 10/10 integration tests pass

## Investigation areas
- Argument marshaling in `redis.call()`: JS Value → byte slices without allocation
- Return value conversion: RESP result → JS Value without intermediate String
- Command dispatch overhead: can we fast-path common commands (GET, SET, INCRBY)?
- String allocation in command name/argument processing
