## Goal
Reduce cost of KEYS/ARGV array setup and array return marshaling to close gap with Lua

## Current state

Benchmarks involving KEYS/ARGV access or array returns are slower than Lua:
- `keys_argv` (return KEYS+ARGV values): 2.00x slower — `return [KEYS[0],KEYS[1],ARGV[0],ARGV[1],ARGV[2]];`
- `lrange` (LPUSH+LRANGE array return): 1.94x slower — returns array from `redis.call('LRANGE',...)`

Lua tables for KEYS/ARGV are lightweight; our JS arrays require full object allocation.
Array return marshaling (JS array → RESP array) likely involves per-element allocation.

## Acceptance Criteria
- [ ] `keys_argv` benchmark ratio drops below 1.5x
- [ ] `lrange` benchmark ratio drops below 1.5x
- [ ] `make lua-js-perf-check` passes gate
- [ ] All cargo tests pass, 10/10 integration tests pass

## Investigation areas
- KEYS/ARGV array creation: pre-allocate with known size, avoid dynamic push
- Array element access: fast-path integer index property gets
- Array return marshaling: iterate JS array → RESP without intermediate Vec<Value>
- Consider lightweight array representation for KEYS/ARGV (read-only, fixed-size)
