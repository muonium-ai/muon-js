# T-000048: Store string values as Arc<[u8]> to avoid GET clones

## Goal

Eliminate the full `Vec<u8>` heap clone on every GET command by storing string values
as `Arc<[u8]>` in the `Value::String` variant, matching `RespValue::Blob(Arc<[u8]>)`.

## Current state

- `store.rs` `Value::String(Vec<u8>)` stores string values as owned `Vec<u8>`.
- `Shard::get_string()` returns `Ok(Some(val.clone()))` — a full heap allocation + memcpy.
- The GET handler in `server.rs` then converts to `RespValue::Blob(v.into())`, creating
  another `Arc<[u8]>` from the cloned Vec.
- For GET (the most common benchmark command), this is two allocations per request.

## Proposed approach

1. Change `Value::String(Vec<u8>)` to `Value::String(Arc<[u8]>)`.
2. Update `get_string()` to return `Arc<[u8]>` via a cheap ref-count bump (`.clone()` on Arc).
3. Update `set_string()` to accept `Arc<[u8]>` or convert `Vec<u8>` to `Arc` at write time.
4. Update `incr_by()` — this modifies the string in-place. Will need to convert back to
   `Vec<u8>` for mutation then re-wrap as `Arc`. Consider keeping a small buffer optimization
   for integer-valued strings.
5. Update all callers in `server.rs` that construct `Value::String(...)`.

## Acceptance criteria

- `Value::String` uses `Arc<[u8]>` internally.
- GET returns `Arc<[u8]>` without heap-cloning the value bytes.
- All 66 tests pass. 10/10 integration tests pass.
- INCR/DECRBY still work correctly (mutation path).

- `cargo test --features mini-redis`
- `bash tests/run_integration.sh`
