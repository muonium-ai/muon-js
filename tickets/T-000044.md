# T-000044: Batch RESP response writes with coalesced flush

## Goal

Coalesce multiple RESP response writes into a single `flush()` call when handling
pipelined commands, reducing syscall overhead. Currently each command response triggers
its own `writer.flush().await`, which results in one `write` syscall per response even
when the client has pipelined multiple commands.

## Current state

- `handle_client()` in `server.rs` reads one command, executes it, writes the response,
  and flushes immediately before reading the next command.
- Under pipelining (`redis-benchmark` default), this means N commands → N flushes → N
  `write` syscalls.
- Redis batches responses and flushes once after processing all available pipelined commands.

## Proposed approach

1. After reading a command, check if more data is available in the `BufReader` buffer
   (via `buffer()` or a non-blocking peek).
2. If more commands are buffered (pipeline), execute and write responses without flushing.
3. When no more commands are immediately available, flush once.
4. Alternative: accumulate responses in a `Vec<u8>` buffer and write+flush in one shot
   after draining all available commands.

## Acceptance criteria

- [ ] AC1: Pipelined commands produce at most one `flush()` per batch.
- [ ] AC2: Non-pipelined (interactive) commands still flush after each response.
- [ ] AC3: `cargo test --features mini-redis` passes (66 tests).
- [ ] AC4: `make mini-redis-benchmark` shows ≥ 10% improvement on pipelined workloads.
