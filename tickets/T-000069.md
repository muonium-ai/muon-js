## Goal
Close the gap on the SET pipelined benchmark where mini-redis is 1.09x slower than Redis.

## Current state

| Test | mini-redis RPS | Redis RPS | Ratio |
|------|---------------|-----------|-------|
| SET | 1,149,425 | 1,250,000 | 1.09x slower |

mini-redis handles ~1.15M SET/s vs Redis at ~1.25M SET/s. The gap is small (9%) and may
be within noise, but worth investigating for easy wins.

## Investigation areas
- **Arc allocation per SET**: Each SET creates `Arc<[u8]>` for value storage — check if
  this allocation is the dominant cost vs Redis's sds string
- **Shard lock contention**: Under pipelined load, shard mutex may cause contention
- **Key hashing cost**: Check if the shard selection hash function is efficient
- **Expiry bookkeeping**: Even when no TTL is set, the expiry code path may have overhead
- **Compare with GET (0.98x)**: GET is at parity, so the write path (insert/replace in
  HashMap + potential expiry cleanup) accounts for the difference
