# T-000052: VM string comparison without heap clone

## Goal

Eliminate unnecessary `Vec<u8>` heap allocations when comparing strings in the bytecode
VM's comparison operators and equality checks.

## Current state

- `vm.rs` OpCode::Lt/Gt/Le/Ge do:
  `ctx.string_bytes(a).map(|b| b.to_vec())` — allocating a new Vec just to compare.
- `vm_loose_eq()` and `vm_strict_eq()` do the same `.to_vec()` pattern.
- These run on every comparison in JS scripts — the hot loop of scripting benchmarks.

## Proposed approach

1. Compare byte slices directly without `.to_vec()`:
   ```rust
   if let (Some(la), Some(lb)) = (ctx.string_bytes(a), ctx.string_bytes(b)) {
       stack.push(JSValue::new_bool(la < lb));
   }
   ```
2. `ctx.string_bytes()` already returns `Option<&[u8]>` — no need to clone.
3. If borrow checker prevents holding two `&[u8]` from ctx simultaneously, copy one
   to a small stack buffer or restructure the calls.

## Acceptance criteria

- VM comparison ops do not allocate `Vec<u8>` for string comparisons.
- All 66 tests pass. All mquickjs compatibility tests pass.
