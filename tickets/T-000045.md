# T-000045: Replace Vec with VecDeque for list storage

## Goal

Replace `Vec` with `VecDeque` for Redis list storage to make LPUSH/LPOP O(1) instead
of O(n). This is the primary cause of the 12x LRANGE performance gap.

## Current state

- `store.rs`: Lists are stored as `Vec<Arc<[u8]>>`.
- `LPUSH` uses `insert(0, ...)` — O(n) because it shifts all existing elements.
- `LPOP` uses `remove(0)` — O(n) for the same reason.
- `LRANGE` accesses elements by index — O(1) per element with Vec, stays O(1) with VecDeque.
- Benchmark: LRANGE_100 is **12.36x slower** than Redis, partly due to O(n) LPUSH building
  the list inefficiently.

## Proposed approach

1. Change the `List` variant in `StoreValue` from `Vec<Arc<[u8]>>` to `VecDeque<Arc<[u8]>>`.
2. Replace `insert(0, ...)` with `push_front()`.
3. Replace `remove(0)` with `pop_front()`.
4. Update `LINDEX`/`LSET`/`LINSERT`/`LREM` to use VecDeque indexing.
5. Update `LRANGE` to use VecDeque's `range()` or index-based access.
6. Update persistence load/save code if it accesses the Vec directly.

## Acceptance criteria

- [ ] AC1: List storage uses `VecDeque` throughout.
- [ ] AC2: LPUSH/LPOP are O(1).
- [ ] AC3: `cargo test --features mini-redis` passes (66 tests).
- [ ] AC4: LRANGE_100 benchmark ratio improves by ≥ 3x (from 12.36x to ≤ 4x).
