## Goal
Improve throughput of the `set_members` scripting benchmark (currently 0.24x vs Redis Lua) by reducing per-invocation overhead in the `redis_call()` C callback path.

## Context
The `set_members` case (`tests/scripting_js_faithful/10_set_members.js`) runs 101 `redis.call()` invocations per script execution (1× DEL + 100× SADD + 1× SMEMBERS). Each call pays the same overhead as described in T-000031. Additionally, SMEMBERS returns a large array (100 elements) which must be converted via `resp_to_js` → `JS_NewArray` + 100× `JS_SetPropertyUint32`, adding significant marshalling cost.

## Acceptance Criteria
- [ ] Reduce `redis_call()` overhead so `set_members` reaches ≥0.45x vs Redis Lua (≈2× current).
- [x] No regression on other scripting cases.
- [x] All existing tests pass.

## Approach Ideas
- Hold DB lock for entire script execution (see T-000031).
- Optimize `resp_to_js` array path: batch array element creation, reduce per-element overhead.
- Fast-path SADD/SMEMBERS dispatch in `redis_call` to avoid full match arm traversal.

## Notes
- T-000031 implemented script-level DB lock, skip_aof, and fast-path dispatch for SADD/SMEMBERS. Result: set_members improved to 0.26x (from 0.24x). Bottleneck is QuickJS interpreter overhead (~17× per-call gap vs LuaJIT). Reaching ≥0.45x requires JS engine improvements.

## Test Plan
- Run 3-round Lua-vs-JS gate before and after.
- Compare `set_members` RPS improvement.
- Verify no regression on other cases.
