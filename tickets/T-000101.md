## Goal

Close the throughput gap on pipelined SET and HSET, where mini-redis is
currently 14–18% slower than Redis 8.6.1.

| Command | mini-redis (rps) | Redis (rps) | Ratio |
|---------|----------------:|------------:|:-----:|
| SET     | 1,356,852       | 1,577,287   | 0.86x |
| HSET    | 1,273,885       | 1,562,500   | 0.82x |

All other pipelined commands (GET, INCR, LPUSH, RPUSH, LPOP, RPOP, SADD) are
at or above Redis parity (1.03x–1.37x).

## Background

Benchmark: `redis-benchmark -c 50 -n 1,000,000 -P 16` (best of 5 runs).

SET and HSET are the only write-heavy commands below parity. Likely causes:
- Lock contention on the shared `Db` store during high-throughput writes
- Allocation overhead in `set_string` / `hash_set` paths (Arc, HashMap insert)
- Value cloning or unnecessary copies on the write path

## Acceptance criteria

- [ ] `make perf-benchmark` shows SET ratio ≥ 0.95x vs Redis
- [ ] `make perf-benchmark` shows HSET ratio ≥ 0.95x vs Redis
- [ ] No regression on other commands (all remain ≥ 0.95x)
- [ ] `make test` passes
- [ ] `make lua-js-perf-check` passes (no scripting regression)

## Test plan

1. Profile SET/HSET hot path with `cargo flamegraph` or `samply`
2. Identify contention points and allocation overhead
3. Implement optimizations
4. `make perf-benchmark` — verify SET ≥ 0.95x, HSET ≥ 0.95x
5. `make test` — all tests pass
6. `make lua-js-perf-check` — scripting gate passes
