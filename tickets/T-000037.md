# T-000037: Property storage: hash map instead of linked list

## Goal

Replace the singly-linked property list on `HeapObject` with a hash map (or flat sorted
array) for objects that exceed a small property count threshold. Currently, every property
read/write is O(n) in the number of properties, which is a major bottleneck for objects
with many properties (environment objects, user objects, etc.).

## Root cause

- `find_prop_value` (context.rs:1129): Linear scan over `prop_head` linked list.
- `set_prop_value` (context.rs:1140): Linear scan to find existing, then linear scan to
  find tail for insertion.
- Environment objects accumulate many properties: `__parent__`, `__var_env__`, `this`,
  `arguments`, all parameters, all locals — easily 10+ properties. Each variable access
  triggers a linear scan.

## Proposed approach

### Option A: Inline + overflow hash map (recommended)

- Keep the linked list for objects with ≤ 8 properties (fast for small objects).
- When `prop_count > 8`, build a `HashMap<(u32, u32), *mut Property>` keyed by
  `(key_kind, key)` for O(1) lookup.
- On mutation, update both the linked list (for iteration order) and the hash map.

### Option B: Flat sorted array

- Store properties in a contiguous `Vec<Property>` sorted by `(key_kind, key)`.
- Binary search for lookup — O(log n).
- Insertion requires shifting; acceptable for objects that are mostly read-heavy.

### Option C: FxHashMap everywhere

- Replace Property linked list entirely with `FxHashMap` (or `ahash`-backed map).
- Simplest implementation but changes iteration order semantics.

## Acceptance criteria

- [ ] AC1: Property lookup for objects with >8 properties is O(1) or O(log n).
- [ ] AC2: Objects with ≤8 properties have no hash map overhead.
- [ ] AC3: Property iteration order preserved (JS spec requires insertion order).
- [ ] AC4: `cargo test --features mini-redis` passes (66 tests).
