## Goal

Fix the pre-existing JS runtime bug where `redis.call()` does not convert
numeric JavaScript arguments to strings before sending them as Redis command
arguments. This causes EVAL scripts that pass numbers (e.g. `0`, `-1`, `100`)
to commands like `LRANGE`, `HSET`, `HGET` to fail with `[object Error]`.

## Background

Discovered during the multi-threaded benchmark (T-000090). The `lrange` and
`hash_sum` cases return errors because the JS scripts pass numeric literals
to `redis.call()`:

```js
// lrange — passes 0 and -1 as numbers
redis.call('LRANGE', KEYS[0], 0, -1);

// hash_sum — passes numeric i and j
redis.call('HSET', KEYS[0], 'f' + i, i);
redis.call('HGET', KEYS[0], 'f' + j);
```

Redis expects all command arguments to be bulk strings. The Lua bridge
auto-coerces numbers via `tostring()`. The JS bridge must do the same.

## Acceptance criteria

- [ ] `redis.call()` / `redis.pcall()` coerce numeric JS values to strings
      before building the RESP command
- [ ] `EVAL` scripts using numeric args (LRANGE with 0/-1, HSET with int
      values, etc.) work correctly
- [ ] MT benchmark `lrange` and `hash_sum` cases produce 0 errors
- [ ] Existing scripting tests still pass
