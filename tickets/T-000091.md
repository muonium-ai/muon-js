## Goal

Fix all JS scripting test scripts to use 1-based KEYS/ARGV indexing, matching
Redis/Lua convention and the server implementation (changed in commit a420b01).

## Background

Discovered during the multi-threaded benchmark (T-000090). The `lrange` and
`hash_sum` cases returned `[object Error]` because the JS test scripts used
0-based indexing (`KEYS[0]`, `ARGV[0]`) while the server creates 1-based
arrays (`KEYS[1]` = first key, `ARGV[1]` = first arg).

Investigation confirmed that numeric argument coercion in `redis.call()` works
correctly (both fast path via `js_value_as_bytes` and slow path via
`js_value_to_arc_bytes` properly convert numbers to strings). The actual root
cause was:

1. `KEYS[0]` resolves to JS `undefined` (slot 0 is empty in 1-based arrays)
2. `undefined` gets silently stringified to key name `"undefined"` in slow path
3. Earlier benchmark cases SET key `"undefined"` as a string type
4. Later cases (lrange, hash_sum) try LPUSH/HSET on key `"undefined"` →
   WRONGTYPE error → thrown as JS exception → serialized as `[object Error]`
