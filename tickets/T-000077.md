## Goal
Make strict equality (===) return false for different types without numeric conversion, per ES spec

## Acceptance Criteria
- [x] `1 === true` returns false (number vs bool, was incorrectly true)
- [x] `0 === false` returns false (number vs bool)
- [x] `null === undefined` returns false (different types)
- [x] `'' === false` returns false (string vs bool)
- [x] Same-type comparisons still work: `1 === 1`, `true === true`, `null === null`
- [x] Loose equality (`==`) still coerces: `1 == true` → true, `null == undefined` → true
- [x] All 72 existing tests pass with zero regressions

## Root Cause
Both `vm_strict_eq` (bytecode VM) and `compare_values` `===` (interpreter) fell through to
`js_to_number` conversion when bit patterns differed. This caused cross-type coercion:
`1 === true` → both convert to `1.0` → true. Per ES spec, `===` must return false when
operand types differ.

## Fix
Added `js_type_tag`/`strict_eq_type_tag` helper that returns a type discriminant
(0=undefined, 1=null, 2=bool, 3=number, 4=string, 5=object). Both code paths now
check type tags before falling through to numeric comparison.

Files changed: `src/vm.rs`, `src/api.rs`, `src/lib.rs` (tests)
