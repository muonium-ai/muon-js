# T-000046: Skip AOF command encoding when persistence is off

## Goal

Avoid building RESP-encoded AOF strings for every mutating command when persistence is
disabled. Currently `build_cmd` clones arguments and encodes them for AOF logging on
every write command, even when no persistence backend is configured.

## Current state

- `server.rs`: After executing mutating commands (SET, INCR, LPUSH, etc.), `build_cmd()`
  is called to construct the RESP representation for AOF logging.
- `build_cmd()` clones the argument vector and formats it as a RESP array.
- When `--persist` is not set, the encoded command is built and then discarded (the
  persist state is a no-op stub).
- This is unnecessary work on every write command.

## Proposed approach

1. Add an `aof_enabled` flag to the server state (set during initialization based on
   whether `--persist` and `--aof` are configured).
2. Guard `build_cmd()` calls with `if aof_enabled { ... }`.
3. Alternatively, make `log_command()` a no-op at the trait level when persistence is off,
   so callers don't need to check.

## Acceptance criteria

- [ ] AC1: No AOF encoding occurs when persistence is not configured.
- [ ] AC2: AOF logging still works correctly when persistence IS configured.
- [ ] AC3: `cargo test --features mini-redis` passes (66 tests).
- [ ] AC4: `make mini-redis-benchmark` (without persistence) shows ≥ 5% improvement on write commands.

## Test plan

- `cargo test --features mini-redis` — all pass.
- `bash tests/run_integration.sh` — 10/10.
- `make mini-redis-benchmark` — compare SET/INCR/LPUSH before/after (no persistence mode).
- Verify AOF still works: run with `--persist /tmp/test.db --aof` and confirm commands are logged.
