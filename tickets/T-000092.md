## Goal
Eliminate the intermediate `Vec` allocation in the string concatenation slow
path. Currently the VM `OpCode::Concat` and `OpCode::Add` (string+string case)
allocate a `Vec`, copy both sides into it, then call `js_new_string_len` which
copies the bytes again into the bump allocator.

## Background
In `src/vm.rs:486` (Concat slow path) and `src/vm.rs:175` (Add string path):
```rust
let mut out = Vec::with_capacity(lb.len() + rb.len());
out.extend_from_slice(lb);
out.extend_from_slice(rb);
stack.push(js_new_string_len(ctx, &out));
```
The context already has `alloc_string_concat_val` for val+bytes. Add an
`alloc_string_concat_bytes` (or similar) that takes two `&[u8]` slices and
copies directly into the bump allocator in a single allocation, skipping the
intermediate Vec entirely.

## Acceptance Criteria
- [ ] New method on Context: `alloc_string_concat_bytes(a: &[u8], b: &[u8]) -> Option<*mut u8>`
- [ ] VM Concat and Add string slow paths use the new method (no intermediate Vec)
- [ ] All tests pass (`make test`)

## Affected Files
- `src/context.rs` — new `alloc_string_concat_bytes` method
- `src/vm.rs` — `OpCode::Concat` slow path, `OpCode::Add` string+string path
