## Goal
Reduce contention from the global `Arc<Mutex<ServerState>>` by introducing finer-grained locking and shorter critical sections in command handling.

## Acceptance Criteria
- [ ] Refactor `ServerState` access so non-overlapping operations on different DBs do not serialize behind one global lock.
- [ ] Keep pub/sub and script cache correctness while narrowing lock scope around mutable shared structures.
- [ ] Preserve transactional semantics (`MULTI`/`EXEC`) and command correctness under concurrent clients.
- [ ] Show improved throughput under concurrent benchmark load vs current baseline.

## Notes
- Hotspot references: repeated `state.lock().await` in `handle_client`, `handle_lrange_fast`, `handle_subscribe`, and `handle_publish` (`src/mini_redis/server.rs`).
- Test plan:
	- Add/extend concurrent benchmark scenario (multiple clients mixed read/write).
	- Run integration tests covering pub/sub and MULTI/EXEC semantics.
	- Compare throughput/latency before/after under same client count.
