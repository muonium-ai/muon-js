# T-000051: Iterative RESP parser to eliminate Box::pin allocations

## Goal

Replace the recursive `read_value` / `read_value_inner` RESP parser with an iterative
state machine to eliminate `Box::pin(async { ... })` heap allocations on every incoming
command frame.

## Current state

- `resp.rs` `read_value()` creates `Box::pin(async move { ... })` per call.
- `read_value_inner()` is called recursively for RESP arrays, each level creating another
  `Box::pin`.
- For standard Redis commands (array of bulk strings), this means 1 + N boxed futures per
  command (1 for the array + N for each bulk string element).
- A typical SET command with 3 args = 4 heap allocations just for parsing.

## Proposed approach

1. RESP protocol arrays in practice only nest one level deep (array of bulk strings).
2. Implement an iterative parser using a simple state machine with a loop:
   - Read type prefix byte
   - If `*`, read count, then loop to read N bulk strings
   - If `$`, read length, read bytes, read CRLF
   - No recursion or boxing needed
3. Keep the recursive path as fallback for deeply nested RESP (rare/never in practice).

## Acceptance criteria

- RESP parsing for standard commands (array of bulk strings) uses no `Box::pin`.
- All 66 tests pass. 10/10 integration tests pass.
- Handles edge cases: inline commands, nested arrays, null bulk strings.
