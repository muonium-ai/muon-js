# T-000036: Reduce allocation in expression evaluation hot path

## Goal

Eliminate heap allocations in the expression evaluation hot path. Currently, many
intermediate operations allocate `String`, `Vec`, or `format!` output on every evaluation.

## Root cause ā€” specific allocation sites

1. **`call_closure_with_this` (parser.rs:1790)**: `body_str = ...to_string()` ā€” allocates
   a fresh String copy of the function body on every function call.

2. **`call_closure_with_this` (parser.rs:1797)**: `func_name = ...to_string()` ā€” allocates
   function name string on every call.

3. **`call_closure_with_this` (parser.rs:1780-1788)**: `ParamSpec` structs with `String`
   name fields ā€” allocated on every call for each parameter.

4. **`eval_expr` compound assignment (api.rs:3160)**: `format!("{} = {} {} {}", ...)` ā€”
   allocates a new string to expand `x += 5` into `x = x + 5`, then recursively evaluates
   the new string.

5. **`strip_comments_checked` (evals.rs:332)**: Allocates a `Vec<u8>` sized to the full
   source, even when the source has no comments.

6. **`normalize_line_continuations` (evals.rs:418)**: Always allocates a new `String`
   output, even when no line continuations exist.

7. **`split_statements` (evals.rs:763)**: Returns `Vec<&str>` ā€” allocates the vector on
   every call. For cached function bodies (T-000034) this is amortized, but for `eval()`
   and top-level code it's not.

8. **`eval_value` arrow function (evals.rs:138)**: `format!("return {};", body_src)` for
   expression-body arrows.

9. **`add_values` string concatenation (api.rs:~10560)**: Creates intermediate `Vec` and
   copies both string byte slices.

## Proposed approach

- **No-comment fast path**: If source contains no `/` character, return a borrowed `&str`
  directly from `strip_comments_checked` instead of allocating.
- **No-normalization fast path**: If source has no line continuations, return borrowed.
- **Compound assignment**: Expand in-place by evaluating `lhs`, computing `lhs op rhs`,
  storing result ā€” no string formatting needed.
- **Borrow function body**: Keep a reference to the heap string instead of `.to_string()`.
- **Pre-allocate param specs**: Use a small stack array (`[ParamSpec; 8]`) for common cases.
- **String concat buffer**: Use a reusable buffer on the context for string concatenation.

## Acceptance criteria

- [ ] AC1: `call_closure_with_this` does not allocate body/name strings if body is cached.
- [ ] AC2: Compound assignment does not use `format!`.
- [ ] AC3: `strip_comments_checked` returns borrowed when no comments present.
- [ ] AC4: `cargo test --features mini-redis` passes (66 tests).
