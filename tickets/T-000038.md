# T-000038: Bytecode compilation and VM execution for hot functions

## Goal

Compile frequently-called functions to bytecode and execute them on the existing `vm.rs`
stack-based VM, instead of re-interpreting source text via the AST-walking evaluator.
This is the single largest potential speedup: a bytecode VM with a tight dispatch loop can
be 10-50× faster than source re-parsing per invocation.

## Current state of the bytecode VM

- `bytecode.rs`: Defines `Instruction`, `OpCode` (26 ops), `BytecodeFunction`,
  `BytecodeModule`. Infrastructure is in place.
- `compiler.rs` (355 lines): `ExprCompiler` can compile arithmetic expressions, comparisons,
  conditionals, logical ops, comma operator, assignments, identifiers, and number literals
  to bytecode. Missing: function calls, property access, string ops, control flow
  (if/while/for), local variables.
- `vm.rs` (173 lines): Stack-based VM with `run_module()`. Handles: Const, Add/Sub/Mul/Div,
  Mod, Neg, Not, comparisons, And/Or, Jump/JumpIfFalse, LoadGlobal/StoreGlobal, Return,
  Drop, Dup. Missing: Call, GetProp, SetProp, NewObject, NewArray, local variable access.

## Proposed approach (incremental)

### Phase 1: Extend compiler for function calls and property access
- Add `OpCode::Call(argc)` — pop function + argc args from stack, call, push result.
- Add `OpCode::GetProp` / `OpCode::SetProp` — pop object + key, push result.
- Add `OpCode::LoadLocal(slot)` / `OpCode::StoreLocal(slot)` for indexed locals (see T-000035).

### Phase 2: Extend compiler for control flow
- `OpCode::JumpIfTrue`, loop constructs via Jump/JumpIfFalse.
- Compile `if`/`while`/`for` statements.

### Phase 3: Compile function bodies
- On first call to a function, attempt to compile its body to bytecode.
- If compilation succeeds, store the `BytecodeFunction` on the function object.
- On subsequent calls, execute via `vm.rs` instead of `eval_function_body`.
- Fall back to interpreter for unsupported constructs (eval, with, etc.).

### Phase 4: Inline caching (future)
- Cache property lookup results for monomorphic call sites.

## Acceptance criteria

- [ ] AC1: Functions using arithmetic, comparisons, variable access, and simple control flow
      compile to bytecode and execute via the VM.
- [ ] AC2: Unsupported constructs gracefully fall back to the interpreter.
- [ ] AC3: `cargo test --features mini-redis` passes (66 tests).
- [ ] AC4: `hash_sum` benchmark improves by ≥ 3× over current (target: ≥ 0.50x of Redis+Lua).

## Test plan

- Unit tests for each new opcode.
- Compile + execute a function with a for-loop summing numbers.
- Benchmark: `python3 scripts/run_lua_js_3rounds.py` before/after.
- Regression: `cargo test` + `bash tests/run_integration.sh`.
