## Goal
Improve throughput of the `hash_sum` scripting benchmark (currently 0.15x vs Redis Lua) by reducing per-invocation overhead in the `redis_call()` C callback path.

## Context
The `hash_sum` case (`tests/scripting_js_faithful/09_hash_sum.js`) runs 200 `redis.call()` invocations per script execution (100× HSET + 100× HGET). Each call pays:
1. JS→C FFI boundary crossing and string marshalling (`JS_ToCString`)
2. Command re-parsing (`parse_command`)
3. Async mutex `block_on(dbs_state[db].lock())` per call
4. Full `handle_command()` dispatch (match on ~50 arms)
5. Response conversion back to JS (`resp_to_js`)

Redis Lua avoids most of this: Lua→C is lightweight, command dispatch is a direct function pointer, and there's no async lock.

## Acceptance Criteria
- [ ] Reduce `redis_call()` overhead so `hash_sum` reaches ≥0.30x vs Redis Lua (2× current).
- [ ] No regression on other scripting cases (hello, keys_argv, redis_call, incrby, lrange).
- [ ] All existing tests pass.

## Approach Ideas
- Hold the DB lock for the entire script execution instead of per-call acquire/release (scripts are atomic in Redis).
- Cache the parsed command string to avoid `parse_command` overhead.
- Use a fast-path lookup table for common commands (HSET, HGET, GET, SET, INCR) instead of the full match dispatch.
- Reduce JS↔Rust string conversion overhead (avoid allocations where possible).

## Test Plan
- Run 3-round Lua-vs-JS gate before and after.
- Compare `hash_sum` RPS improvement.
- Verify no regression on other cases.
