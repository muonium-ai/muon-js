# T-000054: Avoid key.to_vec() for existing keys in store entry API

## Goal

Eliminate wasted `Vec<u8>` key allocations when using HashMap's entry API for keys that
already exist.

## Current state

- Shard methods like `set_nx()`, `hash_set()`, `set_add()`, `list_push()` use:
  `self.data.entry(key.to_vec()).or_insert_with(...)`.
- `key.to_vec()` allocates a new Vec<u8> on every call, even when the key already exists
  (the allocation is wasted and immediately dropped by the entry API).
- For hot keys in benchmarks, this is a significant source of unnecessary allocations.

## Proposed approach

1. For operations that need entry API, check `self.data.contains_key(key)` first:
   - If key exists, use `self.data.get_mut(key).unwrap()` — no allocation.
   - If key doesn't exist, use `self.data.entry(key.to_vec()).or_insert(...)`.
2. Alternatively, use `HashMap::raw_entry_mut()` (nightly) or a crate like `hashbrown`
   which supports `entry_ref()` that can look up by `&[u8]` without allocating.
3. For the simplest approach, the contains_key + get_mut pattern avoids changing deps.

## Acceptance criteria

- Existing-key operations do not allocate a Vec<u8> for the key.
- All 66 tests pass. 10/10 integration tests pass.
