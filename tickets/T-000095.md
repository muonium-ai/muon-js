## Goal
Add memory reclamation to the bump allocator so that GC can actually free dead
objects instead of only marking live ones.

## Background
`MemoryRegion` in `src/context.rs` is a pure bump allocator — its offset only
grows, never shrinks. `gc_collect()` marks live objects but never reclaims dead
ones. This means memory usage monotonically increases until the scripting
runtime's `maybe_reset()` threshold triggers a full context rebuild.

For mini-redis with repeated EVAL calls, this accelerates memory pressure and
forces more frequent full resets (which are expensive — rebuild the entire
JSContext).

## Approach Options
1. **Free list**: Track freed blocks by size class. Reuse on subsequent allocs.
2. **Compaction**: After marking, compact live objects and reset the bump offset.
   Requires updating all pointers (complex but most memory-efficient).
3. **Region-based**: Allocate per-script regions that are freed wholesale
   when the script finishes. Simpler than compaction for the EVAL use case.

Option 1 (free list) is the most pragmatic starting point.

## Acceptance Criteria
- [ ] Dead objects identified by GC are made available for reuse
- [ ] Memory usage for repeated EVAL calls grows sub-linearly
- [ ] `maybe_reset()` triggers are less frequent under scripting workloads
- [ ] All existing tests pass (`make test`)
- [ ] No regressions in parity tests (`make parity-test`)
