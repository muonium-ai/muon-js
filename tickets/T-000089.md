## Goal
Eliminate the unconditional `format!("__get__{}", name)` heap allocation and
extra property lookup that `js_get_property_str()` does on every property read,
and the equivalent `format!("__set__{}", name)` on every property write.

## Background
`js_get_property_str()` in `src/api/mod.rs:466` always constructs a getter key
string via `format!()` and does a full property lookup, even though 99%+ of
objects never have getters/setters defined. Similarly for `js_set_property_str()`
at line 505 with setters. This adds a heap allocation + property chain walk to
every single property access in the runtime.

## Approach
Add a `has_accessors: bool` flag (or bit) on `HeapObject` in `context.rs`.
Set it to `true` when a `__get__*` or `__set__*` property is first defined.
In `js_get_property_str` / `js_set_property_str`, skip the getter/setter
check entirely when the flag is `false` (the overwhelmingly common case).

## Acceptance Criteria
- [ ] `HeapObject` has an accessor flag (bit or bool)
- [ ] Flag is set when `__get__*` or `__set__*` properties are defined
- [ ] `js_get_property_str` skips getter check when flag is false
- [ ] `js_set_property_str` skips setter check when flag is false
- [ ] Getter/setter functionality still works (existing tests pass)
- [ ] All tests pass (`make test`)

## Affected Files
- `src/context.rs` — `HeapObject` struct, `set_prop_value` or `set_property_str`
- `src/api/mod.rs` — `js_get_property_str()`, `js_set_property_str()`
