## Goal
Close the gap on the INCR pipelined benchmark where mini-redis is 1.84x slower than Redis.

## Current state

| Test | mini-redis RPS | Redis RPS | Ratio |
|------|---------------|-----------|-------|
| INCR | 938,967 | 1,724,138 | 1.84x slower |

This is the largest gap across all pipelined benchmarks. mini-redis handles ~939K INCR/s
vs Redis at ~1.72M INCR/s.

## Investigation areas
- **Shard locking overhead**: INCR goes through `shard(key)` which acquires a mutex lock
  per operation — under pipelined load this is high contention
- **String-to-int parse on every INCR**: The stored value is `Arc<[u8]>`, so each INCR must
  parse the string to i64, add, then convert back to string bytes — Redis keeps integers
  in native format
- **Arc allocation on write-back**: Each INCR creates a new `Arc<[u8]>` for the updated value
- **incr_by implementation**: Check if there's unnecessary cloning or allocation in the
  increment path
- **Compare with GET/SET**: GET (0.98x) and SET (1.09x) are near parity, suggesting the
  read/write path itself is fast — the bottleneck is specific to the increment logic
