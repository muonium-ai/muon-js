## Goal
Close the gap on the HSET pipelined benchmark where mini-redis is 1.18x slower than Redis.

## Current state

| Test | mini-redis RPS | Redis RPS | Ratio |
|------|---------------|-----------|-------|
| HSET | 1,226,994 | 1,449,275 | 1.18x slower |

mini-redis handles ~1.23M HSET/s vs Redis at ~1.45M HSET/s. The gap is moderate (18%).

## Investigation areas
- **Hash map implementation**: Check if the hash storage uses an efficient map (HashMap vs
  BTreeMap vs linked list) and whether per-field allocation is excessive
- **Shard lock contention**: Same as INCR — shard mutex under pipelined load
- **Field/value Arc allocation**: Each HSET creates `Arc<[u8]>` for both field and value —
  check if stack-allocated or borrowed alternatives are possible
- **Key lookup overhead**: Hash operations do a key lookup to find the hash, then a field
  lookup within it — two levels of indirection
- **Compare with SET (1.09x)**: SET is closer to Redis, so the extra hash-level indirection
  accounts for part of the gap
