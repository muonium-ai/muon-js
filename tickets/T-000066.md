## Goal
Close the gap on the `hash_sum` scripting benchmark where Lua is 4.0x faster than muonjs.

## Current state

Benchmark ratio: **0.25x** (js_rps / lua_rps) — Lua is ~4.0x faster. This is the largest
gap across all benchmarks.

Script (`09_hash_sum.js`):
```js
redis.call('DEL', KEYS[0]);
var n = Number(ARGV[0] || 100);
for (var i = 1; i <= n; i += 1) {
  redis.call('HSET', KEYS[0], 'f' + i, i);
}
var sum = 0;
for (var j = 1; j <= n; j += 1) {
  var v = redis.call('HGET', KEYS[0], 'f' + j);
  sum += Number(v);
}
return sum;
```

This is the most complex benchmark: two 100-iteration loops (HSET then HGET), string
concatenation (`'f' + i/j`), `Number(v)` coercion on each HGET result, and running sum
accumulation. Total: ~200 redis.call invocations + 100 Number coercions + 200 string concats.

## Investigation areas
- **String concatenation `'f' + i`**: Called 200 times — each creates a new JS string via
  coercion + concat. Profile whether this is a major cost component
- **Number(v) coercion**: Called 100 times on HGET string results — goes through full
  type dispatch. Could fast-path string→number for simple integer strings
- **Two separate loops**: 200 total redis.call dispatches — same per-call overhead as bulk_incr
  but doubled
- **sum += Number(v)**: Binary `+=` with Number coercion — check if this triggers repeated
  float allocation or stays in i32 fast path
- **Variable declaration in loop**: `var v = redis.call(...)` — check if `var` in loop body
  has per-iteration allocation cost
- **HSET/HGET fast paths**: Already exist — verify they're being hit and measure residual overhead
- **Common with T-000064/T-000065**: Loop overhead, string concat, redis.call dispatch are
  shared bottlenecks — fixes to those will improve this benchmark too
