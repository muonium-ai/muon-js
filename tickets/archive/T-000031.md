## Goal
Improve throughput of the `hash_sum` scripting benchmark (currently 0.15x vs Redis Lua) by reducing per-invocation overhead in the `redis_call()` C callback path.

## Context
The `hash_sum` case (`tests/scripting_js_faithful/09_hash_sum.js`) runs 200 `redis.call()` invocations per script execution (100× HSET + 100× HGET). Each call pays:
1. JS→C FFI boundary crossing and string marshalling (`JS_ToCString`)
2. Command re-parsing (`parse_command`)
3. Async mutex `block_on(dbs_state[db].lock())` per call
4. Full `handle_command()` dispatch (match on ~50 arms)
5. Response conversion back to JS (`resp_to_js`)

Redis Lua avoids most of this: Lua→C is lightweight, command dispatch is a direct function pointer, and there's no async lock.

## Acceptance Criteria
- [ ] Reduce `redis_call()` overhead so `hash_sum` reaches ≥0.30x vs Redis Lua (2× current).
- [x] No regression on other scripting cases (hello, keys_argv, redis_call, incrby, lrange).
- [x] All existing tests pass.

## Approach Ideas
- Hold the DB lock for the entire script execution instead of per-call acquire/release (scripts are atomic in Redis).
- Cache the parsed command string to avoid `parse_command` overhead.
- Use a fast-path lookup table for common commands (HSET, HGET, GET, SET, INCR) instead of the full match dispatch.
- Reduce JS↔Rust string conversion overhead (avoid allocations where possible).

## Test Plan
- Run 3-round Lua-vs-JS gate before and after.
- Compare `hash_sum` RPS improvement.
- Verify no regression on other cases.

## Progress Log
- 2026-02-21: Implemented 4 layered optimizations in redis_call path:
  1. Script-level DB lock: hold Mutex<Db> for entire EVAL (eliminates per-call lock acquire/release)
  2. skip_aof flag: handle_command skips persist_state lock when called from scripts (Redis logs EVAL, not inner commands)
  3. Fast-path dispatch: redis_call_fast() bypasses parse_command + handle_command + RespValue for GET, SET, DEL, HSET, HGET, INCR, INCRBY, SADD, SMEMBERS
  4. Stack-allocated command buffer: avoid heap allocation for command string in redis_call
- 2026-02-21: All 66 tests pass. 3-round benchmark: hash_sum=0.16x (baseline 0.15x), no regression on fast cases (hello=2.04x, incrby=1.99x, redis_call=1.91x).
- 2026-02-21: Root-cause finding: the bottleneck is QuickJS interpreter overhead, NOT Rust-side dispatch. Per-call timing: our redis.call costs ~4.87µs vs Redis LuaJIT's ~0.28µs (17× gap). This is the JS bytecode interpreter loop + function call mechanism vs LuaJIT's JIT-compiled native code. No amount of Rust-side optimization can close this gap — reaching ≥0.30x requires JS engine improvements (faster bytecode dispatch, function call inlining, or JIT compilation).
- 2026-02-21: AC1 not met (0.16x vs target 0.30x). AC2-AC3 met. Architectural improvements are correct and will become significant when JS engine speed improves. Marking needs_review for decision on acceptance.
