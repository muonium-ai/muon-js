# T-000058: Fix parity test failures - missing commands and flush bug

## Goal
Fix all pre-existing parity test failures and the test hang. Two root causes:

1. **Missing flush on error continue paths**: When `parse_command` returns `None` (unknown command), the server writes an error response but `continue`s the loop, skipping the `writer.flush()` at the bottom. The response sits buffered until the next successful command triggers a flush. This causes timeouts in the parity test for any unimplemented command.

2. **Missing command implementations**: 10 commands referenced by the parity test are not implemented:
   - **List**: LPUSHX, RPUSHX, LTRIM
   - **Hash**: HINCRBY, HSETNX
   - **Set**: SUNION, SINTER
   - **Stream**: XLEN, XREVRANGE, XDEL

## Acceptance Criteria
- [ ] All `continue` paths in `handle_client` flush the writer when reader buffer is empty
- [ ] LPUSHX returns 0 on missing key, or pushes and returns new length on existing list
- [ ] RPUSHX returns 0 on missing key, or pushes and returns new length on existing list
- [ ] LTRIM trims list to specified range, returns OK
- [ ] HINCRBY increments hash field by integer, returns new value
- [ ] HSETNX sets hash field only if not exists, returns 1/0
- [ ] SUNION returns union of sets
- [ ] SINTER returns intersection of sets
- [ ] XLEN returns stream length
- [ ] XREVRANGE returns stream entries in reverse
- [ ] XDEL removes stream entries by ID
- [ ] All new commands added to `parse_command` HashMap
- [ ] Parity test (`python3 tests/mini_redis_parity.py`) passes all tests without hanging
- [ ] Existing unit and integration tests still pass
