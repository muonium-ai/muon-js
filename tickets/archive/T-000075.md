## Goal
Fix latency p50/p95 metrics showing 0 in the browser demo.

## Root Cause
`CoreExecutor::now_us()` on the WASM path uses `js_sys::Date::now()` which has only ~1ms resolution. Individual mini-redis commands execute in sub-millisecond time, so `start_us` and `end_us` are identical and elapsed latency is always 0.

## Acceptance Criteria
- [x] Replace `js_sys::Date::now()` with `performance.now()` (DOMHighResTimeStamp, microsecond precision) in `now_us()` for the WASM target.
- [ ] Latency p50 and p95 values update to non-zero values during simulation.
- [ ] Queue depth stat updates correctly.

## Fix
Use a `#[wasm_bindgen]` extern binding for `performance.now()`:

```rust
#[cfg(feature = "mini-redis-wasm")]
#[wasm_bindgen]
extern "C" {
    #[wasm_bindgen(js_namespace = performance, js_name = now)]
    fn perf_now() -> f64;
}
```

Then in `now_us()`:
```rust
#[cfg(target_arch = "wasm32")]
{
    (perf_now() * 1_000.0) as u64
}
```

`performance.now()` is available in both main thread and web worker global scopes.
