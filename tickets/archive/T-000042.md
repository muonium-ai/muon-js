# T-000042: Replace Mutex<Db> with sharded locks

## Goal

Replace the single `Mutex<Db>` per database with a sharded locking strategy to reduce
contention under concurrent clients. Currently every command acquires a global mutex on
the entire database, serializing all operations even when they touch different keys.

## Current state

- `server.rs`: `dbs_state: Arc<Vec<Mutex<Db>>>` — one mutex per logical database.
- Every command (GET, SET, INCR, etc.) locks the full `Db` for the entire operation.
- Under pipelining or multiple clients, this creates a serialization bottleneck.
- The benchmark shows 2.6–2.8x slower than Redis; lock contention is a likely contributor.

## Proposed approach

### Option A: Key-sharded DashMap
Replace `HashMap<Vec<u8>, StoreValue>` with `DashMap<Vec<u8>, StoreValue>` (from the
`dashmap` crate). This gives per-shard read/write locks with no global serialization.

### Option B: Manual shard buckets
Partition the keyspace into N buckets (e.g., 64), each with its own `RwLock`. Hash the
key to select a bucket. This avoids adding a new dependency.

### Considerations
- `MULTI`/`EXEC` transactions need atomic access to multiple keys — may need a fallback
  to full lock for transaction mode.
- `KEYS`/`SCAN`/`FLUSHDB` need to iterate all shards.
- `EVAL` scripts hold the lock for atomic execution — sharding must preserve this guarantee.

- [ ] AC2: `MULTI`/`EXEC` maintains atomicity across multiple keys.
