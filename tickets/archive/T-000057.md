# T-000057: Lock-free AOF channel to replace tokio::spawn per command

## Goal

Replace the current `tokio::spawn` fire-and-forget pattern for AOF logging with a more
efficient mechanism that avoids per-command task spawn overhead and argument cloning.

## Current state

- `server.rs` `log_cmd!` macro (when AOF is enabled) does:
  1. `$args.to_vec()` — clones all `Arc<[u8]>` arguments
  2. `$cmd.to_string()` — allocates a String for the command name
  3. `Arc::clone(persist_state)` — bumps ref count
  4. `tokio::spawn(async move { ... })` — spawns a new tokio task
- Each of these has overhead, and tokio::spawn in particular adds scheduling cost.
- This runs on every mutating command when persistence + AOF is enabled.

## Proposed approach

1. Use a bounded lock-free MPSC channel (e.g., `crossbeam::channel::bounded`) instead
   of tokio::spawn + mpsc.
2. Pre-encode the RESP command bytes on the caller side into a single `Vec<u8>` buffer,
   then send the buffer through the channel (one allocation instead of N+1).
3. The AOF worker thread reads from the channel and writes batches to disk.
4. Consider using `try_send` with backpressure (drop or block) to avoid unbounded memory.

## Acceptance criteria

- AOF logging does not spawn a tokio task per mutating command.
- Argument cloning is minimized (ideally one buffer per command).
- All 66 tests pass. 10/10 integration tests pass.
- AOF correctness maintained (commands logged in order, batched writes flushed).
