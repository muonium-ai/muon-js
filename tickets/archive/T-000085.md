## Goal
Replace DELETE-all + INSERT pattern in persist.rs snapshot with incremental or INSERT OR REPLACE approach to reduce I/O and memory during save

## Acceptance Criteria
- [x] String/Int kv rows use INSERT OR REPLACE (upsert) instead of DELETE-all + INSERT
- [x] Collection types (list, set, hash, zset, stream) use per-key DELETE + re-insert for their items
- [x] Stale keys deleted from all tables after upsert phase
- [x] All existing tests pass
- [x] No warnings

## Root Cause
`snapshot_inner` wiped all 6 tables (`DELETE FROM kv`, etc.) then re-inserted every row. For large datasets where most data is unchanged between snapshots, this caused unnecessary I/O — every row was deleted and rewritten even if unchanged.

## Fix
- Changed `INSERT` to `INSERT OR REPLACE` for the `kv` table (handles string/int updates in-place)
- For collection types, replaced global `DELETE FROM <table>` with per-key `DELETE FROM <table> WHERE db=? AND key=?` before re-inserting items
- After upserting all live data, query stored keys per-db and delete any stale keys not in the current snapshot
- Net effect: unchanged string keys are updated in-place (SQLite no-ops for identical data), only modified/deleted keys trigger I/O
