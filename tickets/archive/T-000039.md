# T-000039: Builtin method dispatch via jump table instead of string matching

## Goal

Replace the long chains of string comparisons in `parse_postfix` (api.rs:8707) and
`call_builtin_method` (api.rs:8965) with a hash map or perfect-hash jump table.
Currently, every `.method()` access on strings, arrays, Math, JSON, etc. walks through
dozens of `if marker == "__builtin_..."` branches.

## Root cause

- `parse_postfix` `.` handler (api.rs:8720-8910): When accessing a property on a builtin
  type, the code checks `marker == "__builtin_Math__"`, then matches `prop` against 20+
  entries. For arrays: 17 entries. For strings: 25+ entries. These are serial string
  comparisons.

- `call_builtin_method` (api.rs:8965-end): Another cascade of `if marker == "..."` for
  dispatching the actual method call. Combined, a single method call like `arr.push(x)`
  goes through: property access dispatch (string compare × type × methods) + method call
  dispatch (string compare × all builtins).

- Each comparison involves: `ctx.string_bytes()` → `from_utf8()` → `marker.to_string()` →
  then matching. The `to_string()` at line 8972 allocates on every call.

## Proposed approach

1. **Intern builtin marker atoms at context creation**. Assign each builtin marker
   (`__builtin_Math_floor__`, etc.) a unique numeric ID at startup.
2. **Jump table dispatch**: In `parse_postfix` and `call_builtin_method`, compare the
   atom ID (integer) instead of the string content. Use a `match` on the atom ID, which
   Rust compiles to a jump table.
3. **Eliminate `to_string()` in `call_builtin_method`**: Use byte comparison directly,
   or compare atom IDs.
4. **Static perfect hash**: For the ~100 builtin markers, generate a perfect hash at
   compile time (e.g., using `phf` crate or a hand-rolled minimal hash).

### Quick win (no dependency)

Even without a perfect hash, replacing the cascading `if marker == "..."` with a single
`match` on a pre-computed hash of the marker string would be a significant improvement.
