# T-000035: Scope chain as indexed slots instead of property-linked objects

## Goal

Variable lookup is currently O(properties × scope depth). Every `resolve_binding()` call
(context.rs:264) walks a linked list of environment objects, each connected via
`__parent__` properties. Within each environment, `has_property_str()` interns the name
string and walks a linked `Property` list. For deeply nested closures or tight loops,
this is extremely expensive.

## Root cause

- **Environment objects are JS objects** with properties stored as a singly-linked list.
- Each scope level adds an object linked via `__parent__` property lookup.
- `find_prop_value` (context.rs:1129) is a linear scan over the property linked list.
- Variable names are re-interned (`intern_string`) on every access.
- Special properties like `__parent__`, `__var_env__`, `__body__`, `__params__`,
  `__closure__`, `__env__`, `this`, `arguments` pollute the same property namespace.

## Proposed approach

1. **Indexed local slots**: At function entry, assign each parameter and local variable a
   numeric slot index (0, 1, 2, …). Store values in a `Vec<JSValue>` on the call frame
   instead of setting them as named properties.
2. **Compile-time slot resolution**: When the function body is first parsed (see T-000034),
   scan for all `var`/`let`/`const` declarations and parameter names. Build a name→index
   map. During expression evaluation, resolve known locals by index instead of string lookup.
3. **Fast parent chain**: Replace `__parent__` property lookup with a direct `parent: Value`
   field on the environment, or store the parent index in the call frame stack.
4. **Fallback**: Unknown/dynamic names (e.g. `eval`-injected) fall back to the current
   property-based lookup.

### Incremental path

Phase 1 can be scoped to just parameters + top-level locals, which covers the majority of
variable accesses in typical scripts.
