## Goal
Replace mpsc::unbounded_channel in pub/sub with bounded channels to add backpressure for slow subscribers

## Acceptance Criteria
- [x] Pub/sub uses bounded channels with a configurable capacity
- [x] Slow subscribers have messages dropped instead of unbounded queue growth
- [x] Publishers are not blocked (use try_send, not async send)
- [x] All existing tests pass

## Root Cause
Each client connection created an `mpsc::unbounded_channel` for pub/sub message delivery. A slow subscriber (or one that stops reading) would cause unbounded memory growth as messages accumulated in the channel with no backpressure.

## Fix
Replaced `mpsc::unbounded_channel` with `mpsc::channel(1024)` bounded channel. Changed `UnboundedSender` to `Sender` in the PubSubState type and `handle_subscribe`. Changed `send()` to `try_send()` in `handle_publish` so publishers are never blocked — messages to full channels are simply dropped, matching Redis's fire-and-forget pub/sub semantics.
