## Goal
Close the gap on the `bulk_incr` scripting benchmark where Lua is 3.2x faster than muonjs.

## Current state

Benchmark ratio: **0.31x** (js_rps / lua_rps) — Lua is ~3.2x faster.

Script (`11_bulk_incr.js`):
```js
redis.call('SET', KEYS[0], 0);
var n = Number(ARGV[0] || 100);
for (var i = 0; i < n; i += 1) {
  redis.call('INCRBY', KEYS[0], 1);
}
return redis.call('GET', KEYS[0]);
```

This is the most loop-intensive benchmark: 100 iterations of `redis.call('INCRBY', ...)` plus
a SET at the start and GET at the end. The bottleneck is per-iteration overhead since each
INCRBY call already has a fast path.

## Investigation areas
- **Per-iteration JS overhead**: Each loop iteration involves variable lookup for `i`, comparison
  `i < n`, increment `i += 1`, and a `redis.call()` dispatch — cumulative overhead is ~100x
- **For-loop implementation**: Check if the for-loop body is re-parsed each iteration vs cached
- **Variable access speed**: `KEYS[0]` is accessed 102 times — check if array element access
  is being optimized on repeated access
- **Literal argument handling**: `redis.call('INCRBY', KEYS[0], 1)` — the `1` is a literal
  integer; verify it takes the int fast path and doesn't allocate
- **redis.call dispatch per iteration**: Even with fast paths, ~100 function calls through the
  C function bridge + command matching adds up
- **Consider**: batch/pipeline optimization for same-command loops (speculative)
