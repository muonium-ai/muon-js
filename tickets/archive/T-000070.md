## Goal
Improve INCR pipelined and bulk_incr scripting performance by reducing increment-path parse/allocation overhead

## Acceptance Criteria
- [x] `INCR` pipelined benchmark improves materially versus previous verification (`2.02x` slower baseline)
- [x] `bulk_incr` median ratio improves versus previous verification (`0.33x` baseline)
- [x] `cargo test --features mini-redis` passes
- [x] `bash tests/run_integration.sh` passes (10/10)

## Notes
- Shared bottleneck was `Db::incr_by`: repeated parse of decimal bytes and fresh allocation/write-back on every increment.
- Implemented internal integer representation for incremented string keys to remove parse+alloc from repeated `INCR/INCRBY`.
- Kept external Redis semantics by converting integer-backed values to string bytes at string command boundaries (`GET`/`APPEND`/snapshot).

## Current state (2026-02-22 verification)

  - mini-redis: `1,869,158.88 rps`
  - redis: `1,030,927.81 rps`
  - ratio (`redis/mini`): `0.55x` (previous verified baseline: `2.02x`)
