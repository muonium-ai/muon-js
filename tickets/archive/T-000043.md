# T-000043: Evaluate tokio with io_uring for async I/O

## Goal

Replace `async-std` with `tokio` as the async runtime to gain access to its more
optimized I/O driver, work-stealing scheduler, and future `io_uring` support on Linux.
Benchmark the impact on throughput.

## Current state

- `async-std 1.12` used throughout: `TcpListener`, `TcpStream`, `BufReader`/`BufWriter`,
  `Mutex`, `channel`, `task::spawn`.
- async-std's executor is less optimized than tokio's multi-threaded work-stealing scheduler.
- On Linux, tokio can use `io_uring` via `tokio-uring` for zero-copy I/O.
- CPU profiling showed scheduler overhead (`async_global_executor`, `async-io::reactor`).

## Proposed approach

1. Replace `async-std` dependency with `tokio` (features: `net`, `sync`, `rt-multi-thread`, `macros`).
2. Migrate imports:
   - `async_std::net::TcpListener/TcpStream` → `tokio::net::*`
   - `async_std::io::BufReader/BufWriter` → `tokio::io::*`
   - `async_std::sync::Mutex` → `tokio::sync::Mutex`
   - `async_std::channel` → `tokio::sync::mpsc`
   - `async_std::task::spawn` → `tokio::spawn`
3. Replace `#[async_std::main]` with `#[tokio::main]`.
4. Fix `block_on` calls in scripting path (tokio panics on nested `block_on`).
5. Benchmark before/after on macOS and Linux.
